================================================================================
          TERJUMAN.LIVE — SECURE REMOTE INTERPRETATION GATEWAY
                      SYSTEM ARCHITECTURE & DESIGN DOCUMENT
================================================================================

This document outlines the detailed system specification, data models, API endpoints,
and a comprehensive production-ready prompt to generate a Django-based REST API
backend for Terjuman.live (formerly known as Elliot Remote Interpretation Portal).

--------------------------------------------------------------------------------
1. ARCHITECTURAL OVERVIEW
--------------------------------------------------------------------------------
The Terjuman.live application is a highly responsive, remote real-time 
interpretation platform supporting AI, Human, and Hybrid interpretation modes. It 
specializes in local East African languages (Amharic, Afaan Oromo, Somali, Tigrinya, 
Afar) and English.

Frontend:
- Single-page application (SPA) built with React 18, Vite, TypeScript, Tailwind CSS, 
  Framer Motion, and Lucide Icons.
- Live dashboards for three separate user roles:
  * Client: Requests instant or scheduled interpretation, manages SLA contracts, and
    maintains a real-time digital wallet.
  * Interpreter: Manages availability schedules, receives incoming translation calls,
    assists in live translation chat/audio, and manages earnings.
  * Admin: Monitors live active sessions, reviews detailed audit logs, handles SLA
    approvals, and performs emergency overrides.
- Poller syncing mechanism that polls `/api/init` to synchronize application state.

Backend (To be built in Django):
- Handles authentication and user role-based session management.
- Exposes RESTful API endpoints matching the frontend's expected format.
- Integrates with the Google Gemini API to handle translation matching, smart-routing
  interpreters, speech caption simulation, and session summarization.
- Persists all session transcripts, wallets, audit logs, and availability structures.

2. DATA MODELS & RELATIONAL SCHEMA (DJANGO DESIGN)
--------------------------------------------------------------------------------

A robust Django backend will require the following database tables (represented 
conceptually as Django models):

A. Profile (One-to-One with django.contrib.auth.models.User)
   - user: OneToOneField(User)
   - role: CharField (Choices: 'admin', 'client', 'interpreter')
   - status: CharField (Choices: 'active', 'suspended', 'pending')
   - languages: ArrayField / JSONField (e.g., ["Amharic", "English"])
   - rating: FloatField (Default: 5.0)
   - completed_sessions: IntegerField (Default: 0)
   - hourly_rate: DecimalField / FloatField (Default: 40.0)
   - avatar: URLField (Optional profile image)

B. ContractDetails (SLA Agreements)
   - contract_id: CharField (Primary Key / UUID / Code)
   - organization_name: CharField (e.g., "Ethiopian Ministry of Health")
   - signed_date: DateTimeField
   - expiry_date: DateTimeField
   - sla_level: CharField (e.g., "Tier-1 Healthcare Gold SLA")
   - billing_code: CharField
   - max_concurrent_sessions: IntegerField (Default: 5)
   - status: CharField (Choices: 'active', 'expired')

C. Session (Interpretation Sessions)
   - id: CharField (Primary Key, e.g., 'sess_1024')
   - client: ForeignKey(User, related_name='client_sessions')
   - client_name: CharField (Denormalized for instant UI binding)
   - interpreter: ForeignKey(User, related_name='interpreter_sessions', null=True)
   - interpreter_name: CharField (Optional, denormalized)
   - language_from: CharField (e.g., "Amharic")
   - language_to: CharField (e.g., "Afaan Oromo")
   - service_type: CharField (Choices: 'medical', 'legal', 'business', 'general')
   - service_mode: CharField (Choices: 'AI', 'Human', 'Both')
   - status: CharField (Choices: 'pending', 'incoming', 'active', 'completed', 'cancelled', 'missed')
   - scheduled_time: CharField / DateTimeField (null for instant, or ISO string)
   - cost: FloatField (Default: 0.0)
   - duration_seconds: IntegerField (Default: 0)
   - transcript: JSONField (List of strings representing dialogue transcript)
   - summary: TextField (AI-generated session wrap-up)
   - rating_by_client: IntegerField (Optional, 1 to 5)
   - review_by_client: TextField (Optional feedback text)
   - emergency_triggered: BooleanField (Default: False)

D. ChatMessage (Live Session Messages)
   - session: ForeignKey(Session, related_name='chat_messages')
   - sender_role: CharField (Choices: 'client', 'interpreter', 'system')
   - sender_name: CharField
   - text: TextField
   - translated_text: TextField (Optional, generated by AI)
   - timestamp: DateTimeField (Auto_now_add=True)

E. Transaction (Digital Wallet Statements)
   - id: CharField (Primary Key, e.g., 'tx_ch99221')
   - user: ForeignKey(User)
   - user_name: CharField (Denormalized)
   - type: CharField (Choices: 'deposit', 'payment', 'payout', 'refund')
   - amount: FloatField
   - status: CharField (Choices: 'completed', 'pending', 'failed')
   - timestamp: DateTimeField (Auto_now_add=True)
   - reference: CharField (e.g., 'CHP-MOCK-99221')

F. InterpreterAvailability
   - interpreter: ForeignKey(User)
   - day: CharField (e.g., "Monday", "Tuesday", etc.)
   - slots: JSONField (List of objects: { start: string, end: string, recurring: boolean })

G. AuditLog
   - id: CharField (Primary Key)
   - timestamp: DateTimeField (Auto_now_add=True)
   - action: CharField (e.g., "Admin System Booted")
   - user_role: CharField
   - user_name: CharField
   - status: CharField (Choices: 'info', 'success', 'warning', 'danger')

3. DETAILED ENDPOINT SPECIFICATIONS
--------------------------------------------------------------------------------

The frontend integrates via fetch queries to relative paths. All API paths return JSON:

---
GET /api/init
- Purpose: Retrieve the entire application state (users, sessions, transactions,
           availabilities, active contract state, audit logs) for the logged-in context.
- Returns:
  {
    "users": [...],
    "sessions": [...],
    "transactions": [...],
    "availabilities": [...],
    "contractsList": [...],
    "activeContractId": "ELLIOT-CON-MOH-2026",
    "aiAvailable": true,
    "clientWalletBalance": 2450.0,
    "auditLogs": [...]
  }

---
POST /api/sessions/request
- Purpose: Request a live human interpreter or launch an automated AI session.
- Payload:
  {
    "languageFrom": "Amharic",
    "languageTo": "English",
    "serviceType": "medical",
    "serviceMode": "AI" | "Human" | "Both",
    "scheduledTime": "instant" | ISO string
  }
- Returns:
  {
    "success": true,
    "session": { ... }
  }

---
POST /api/sessions/<id>/accept
- Purpose: Interpreter accepts a pending call invitation.
- Payload:
  {
    "interpreterId": "usr_int1",
    "interpreterName": "Bekele Megersa"
  }
- Returns:
  {
    "success": true,
    "session": { ... }
  }

---
POST /api/sessions/<id>/reject
- Purpose: Interpreter declines/rejects a pending session request.
- Payload:
  {
    "interpreterId": "usr_int1"
  }
- Returns:
  {
    "success": true,
    "session": { ... }
  }

---
POST /api/sessions/<id>/chat
- Purpose: Add a message in the active session's chat log, optionally translating it using Gemini.
- Payload:
  {
    "senderRole": "client" | "interpreter",
    "senderName": "Dawit Yohannes",
    "text": "Hello, I need an Amharic translator."
  }
- Returns:
  {
    "success": true,
    "message": { ... },
    "session": { ... }
  }

---
POST /api/sessions/<id>/complete
- Purpose: Terminate the active call, log calculated session cost, deduct from client, payout interpreter.
- Payload:
  {
    "durationSeconds": 120,
    "rating": 5,
    "review": "Awesome experience!"
  }
- Returns:
  {
    "success": true,
    "session": { ... },
    "clientWalletBalance": 2435.0
  }

---
POST /api/sessions/<id>/intervene
- Purpose: Admin intervenes in a critical session, triggering emergency translation bypass.
- Payload: {}
- Returns:
  {
    "success": true,
    "session": { ... }
  }

---
POST /api/wallet/deposit
- Purpose: Add funds to a client's digital wallet.
- Payload:
  {
    "amount": 500,
    "reference": "CHP-99281"
  }
- Returns:
  {
    "success": true,
    "amount": 500,
    "balance": 2950.0
  }

---
POST /api/wallet/payout
- Purpose: Transfer earned funds to an interpreter's external mobile bank.
- Payload:
  {
    "amount": 250,
    "interpreterId": "usr_int1"
  }

  {
    "success": true,

POST /api/contract/create

    "organizationName": "Ministry of Justice",

4. PROMPT FOR GENERATING THE DJANGO BACKEND
--------------------------------------------------------------------------------

Copy and paste the following prompt to generate your Django backend using any
advanced AI coder (such as Gemini in Google AI Studio or your local pipeline):

\"\"\"
Generate a complete, production-ready Django application that exposes a secure, high-performance REST API backend for the Terjuman.live remote translation portal.

The backend must fulfill the following technical constraints:
1. Framework: Django 5.x + Django REST Framework (DRF).
2. Database: PostgreSQL-compatible (default to sqlite3 for development with clean instructions to switch to PostgreSQL/Cloud SQL).
3. Models & Schemas:
   - Profile: Links to django.contrib.auth.models.User. Stores 'role' (admin, client, interpreter), 'status' (active, suspended, pending), 'languages' (list of strings), 'rating' (float), 'completed_sessions' (int), 'hourly_rate' (decimal), 'avatar' (URL string).
   - ContractDetails: Tracks SLAs (contract_id, organization_name, signed_date, expiry_date, sla_level, billing_code, max_concurrent_sessions, status).
   - Session: Tracks translation sessions (id, client, client_name, interpreter, interpreter_name, language_from, language_to, service_type, service_mode, status, scheduled_time, cost, duration_seconds, transcript [JSON list], summary [TextField], rating_by_client, review_by_client, emergency_triggered).
   - ChatMessage: (session, sender_role, sender_name, text, translated_text, timestamp).
   - Transaction: (id, user, user_name, type [deposit, payment, payout, refund], amount, status, timestamp, reference).
   - InterpreterAvailability: (interpreter, day, slots [JSON list of {start, end, recurring}]).
   - AuditLog: (id, timestamp, action, user_role, user_name, status).

4. API Endpoints Required:
   - GET /api/init: Aggregates and returns the entire database state (serialized Users, Sessions, Transactions, Availabilities, Contracts, Active SLA Contract ID, Wallet balances, AuditLogs).
   - POST /api/sessions/request: Creates a session. If mode is "AI", automatically simulate AI responses or initialize the session directly.
   - POST /api/sessions/<id>/accept: Assigns an interpreter to the session and flags it 'active'.
   - POST /api/sessions/<id>/reject: Rejects a session, updating its invitation status.
   - POST /api/sessions/<id>/chat: Creates a ChatMessage. If serviceMode is 'AI' or 'Both', trigger an automatic call to the Google Gemini API to translate the text from `language_from` to `language_to` and populate `translated_text`.
   - POST /api/sessions/<id>/complete: Marks the session complete, records the final duration/rating/review, computes the total cost (duration in hours * interpreter's hourly rate), deducts it from the Client's wallet balance, creates a Payment transaction, pays out the interpreter (creating a Payout transaction), and writes a detailed audit log.
   - POST /api/sessions/<id>/intervene: Admin-override to trigger an emergency bypass.
   - POST /api/wallet/deposit: Increases client balance, creates transaction.
   - POST /api/wallet/payout: Simulates instant payout to mobile banking.
   - POST /api/contract/create: Creates a new contract detail.
   - POST /api/contract/select: Set active contract.
   - POST /api/contract/extend: Adds 30 days to contract expiry.
   - POST /api/users/<id>/update: Modifies languages or rate on profile.

5. AI & Gemini Integration (Using google-genai SDK or direct HTTPS calls):
   - Integrates with the Google Gemini model (e.g., gemini-2.5-flash) using an environment variable `GEMINI_API_KEY`.
   - Implement these Python services or utils:
     - `translate_text(text, from_lang, to_lang, context)`: Returns context-aware professional translation.
     - `get_speech_caption(from_lang, to_lang, type)`: Generates simulated transcript text representing typical medical, legal, or business dialogues.
     - `summarize_session_transcript(transcript_list)`: Creates an executive bulleted summary of a transcript.
     - `smart_match_interpreter(from_lang, to_lang, profiles_list)`: Analyzes available interpreter profiles to rank and match the best candidate using AI weights.

6. Configuration & Security:
   - Support CORS (django-cors-headers) to allow frontend connections.
   - Set up API routes cleanly in `api/urls.py`.
   - Include initial database seeding (using Django migrations or fixtures) so standard demo users (Almaz, Dawit, Bekele, Haleema, Yared, Selamawit) are seeded with their correct roles, rates, and historical logs.
   - Structure the project cleanly with settings, apps, serializers, views, and services.
\"\"\"

5. PREPARING THE FRONTEND SYSTEM FOR THE BACKEND
--------------------------------------------------------------------------------
The current React system is fully decoupled and ready to bind directly to this Django 
backend:
1. Relative Fetch Routes: The frontend queries relative routes like `/api/init` and 
   `/api/sessions/...`. This means once the Django server is configured, you can 
   either:
   - Configure Vite's proxy in `vite.config.ts` to forward `/api` to Django's host during development:
     server: {
       proxy: {
         '/api': 'http://localhost:8000'
       }
     }
   - Or configure Django to serve the React built assets from the `dist/` folder directly in production, exactly like the custom Express server does.
2. Complete Data Contracts: The interfaces declared in `src/types.ts` correspond 
   1:1 to the keys returned in the JSON structures of Django REST framework serializers.
3. Clean Global Polling: The background poller inside `src/App.tsx` handles starting states and connection timeouts gracefully.
